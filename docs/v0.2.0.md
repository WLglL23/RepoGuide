## 概述与动机

在 v0 阶段，RepoGuide 的数据流由多个松散字典组成：`scan_repo()` 返回 `list[dict]`，`classify_files()` 再次返回 `list[dict]`，`generate_project_map()` 返回 `dict`，`format_project_map()` 再消费这个 dict。这种设计在快速原型阶段可行，但存在以下问题：

- **缺乏类型安全**：到处传递原始字典，字段名容易拼写错误，IDE 无法有效提示。
- **逻辑分散**：语言推断、角色映射、便捷列表构建等逻辑散落在不同函数中，修改一处可能忘记另一处。
- **难以扩展**：后续要加入索引、诊断、LLM 问答等功能时，找不到稳定的核心模型作为载体。
- **测试困难**：每个环节都需要手动组装字典，边界条件容易遗漏。

v1 Core SDK 的目标是**在不破坏现有功能的前提下，逐步将字典替换为结构化数据模型**，为未来的 CLI、API 和高级分析能力建立坚实基础。

**当前版本只处理静态扫描信息，不涉及 AST 解析、LLM、索引持久化或 Web 服务。**

## 核心数据模型

所有核心数据模型均使用 Python 的 `dataclass` 定义，字段带有类型注解和详细文档字符串。

### 2.1 RepoFile

**文件位置**：`repoguide/core/models/repo_file.py`

`RepoFile` 表示仓库中的**单个文件**，封装了文件系统扫描和分类器产生的所有基础信息。

#### 字段说明

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `path` | `str` | scanner | 文件相对于仓库根目录的路径。 |
| `name` | `str` | scanner | 文件名（含扩展名），由 `path` 自动提取。 |
| `size` | `int` | scanner | 文件大小（字节）。 |
| `extension` | `str` | scanner | 扩展名（小写，包含点号），如 `".py"`。 |
| `modified_time` | `float` | scanner | 最后修改时间（Unix 时间戳）。 |
| `role` | `str` | classifier | 文件角色，如 `"entrypoint_candidate"`、`"config"`、`"readme"` 等。 |
| `language` | `str` | 自动推断 | 编程/标记语言，默认由 `LanguageDetector` 根据扩展名推断。 |

#### 关键方法

- **`__post_init__()`**：自动规范化字段并对 `language` 进行缺省填充。  
- **`from_dict(data: Dict) -> RepoFile`**：从旧的字典格式（v0 输出）构造实例，保证向后兼容。  
- **`to_dict() -> Dict`**：转换回字典，供旧 `formatter` 或 `ProjectMapper` 使用。  
- **`guess_language(extension: str) -> str`**（静态方法）：根据扩展名返回语言名称。  

#### 设计理由

- 将分散的文件信息收敛到一个对象中，便于传递和校验。
- `language` 字段不在构造时强制传入，而是自动补全，减少调用方的心智负担。
- 提供 `from_dict`/`to_dict` 桥接方法，使 v0 和 v1 可以并行运行。

### 2.2 RepoSnapshot

**文件位置**：`repoguide/core/models/repo_snapshot.py`

`RepoSnapshot` 表示**一次完整仓库扫描的结构化快照**，是 v1 的核心中间聚合对象。

#### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `repo_id` | `str` | 仓库唯一标识（如 `"local:/abs/path"`）。 |
| `root_path` | `str` | 本地根路径。 |
| `project_name` | `str` | 项目名称。 |
| `project_type` | `str` | 项目类型（由 `identify_project_type` 提供）。 |
| `files` | `List[RepoFile]` | 所有文件的 RepoFile 列表。 |
| `file_count` | `int` | 文件总数（自动修正为 `len(files)`）。 |
| `created_at` | `str` | 快照创建时间（ISO 8601 UTC）。 |
| `entrypoints` | `List[str]` | 入口文件路径列表（自动填充）。 |
| `config_files` | `List[str]` | 配置文件路径列表。 |
| `build_files` | `List[str]` | 构建文件路径列表。 |
| `test_files` | `List[str]` | 测试文件路径列表。 |
| `important_files` | `List[str]` | 重要文件路径列表（包含 readme、配置、构建、入口等）。 |

#### 关键行为

- **`__post_init__()`**：如果便捷列表为空，则根据 `files` 中各个 `role` 字段自动提取并去重排序。
- **`to_dict() -> Dict`**：转换为兼容旧流程的大字典。
- **`files_as_dicts() -> List[Dict]`**：仅返回文件列表的字典形式，方便旧 `ProjectMapper` 复用。

#### 设计理由

- 在扫描和分类之后，立即将所有信息冻结为一个不可变风格的对象，下游只读不写。
- 自动派生的便捷列表避免了手动维护多个列表的麻烦，且保证了数据一致性。
- 为将来扩展 **索引（Index）**、**诊断（Diagnostic）** 等模块提供了清晰的输入。

### 2.3 ProjectMap

**文件位置**：`repoguide/core/models/project_map.py`

`ProjectMap` 是从 `RepoSnapshot` 聚合得到的**面向展示层的高层结构视图**，直接供 formatter、CLI、API 等消费。

#### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `root_path` | `str` | 项目根路径。 |
| `project_type` | `str` | 项目类型。 |
| `file_count` | `int` | 文件总数。 |
| `important_files` | `List[str]` | 高价值文件列表。 |
| `entrypoint_candidates` | `List[str]` | 入口候选列表（兼容 v0 命名）。 |
| `config_candidates` | `List[str]` | 配置候选列表。 |
| `build_files` | `List[str]` | 构建文件列表。 |
| `test_files` | `List[str]` | 测试文件列表。 |
| `possible_run_commands` | `List[str]` | 建议的运行/测试命令。 |
| `top_level_tree` | `str` | 一级目录树的文本表示。 |
| `repo_id` | `str` | 仓库标识（v1 新增，默认空）。 |
| `project_name` | `str` | 项目名称（v1 新增，默认空）。 |
| `generated_at` | `str` | 生成时间戳。 |
| `language_breakdown` | `Dict[str, int]` | 各语言文件数量统计。 |
| `role_summary` | `Dict[str, int]` | 各角色文件数量统计。 |
| `top_languages` | `List[str]` | 文件数最多的前几种语言。 |
| `metadata` | `Dict` | 扩展元数据。 |

#### 关键方法

- **`to_dict() -> Dict`**：输出与 v0 格式完全兼容的字典，确保旧 formatter 可以直接使用。

#### 设计理由

- 字段命名保持与 v0 一致（如 `entrypoint_candidates` 而不是 `entrypoints`），避免修改 formatter。
- 新增统计字段为未来的分析报告提供数据，而不会干扰现有功能。
- 仍然是一个纯数据对象，不包含业务逻辑。

## 语言检测模块

**文件位置**：`repoguide/core/language/language_detector.py`

### LanguageDetector

基于文件扩展名的静态语言检测器，第一版仅支持最常见的编程和标记语言。

#### 支持的语言映射

| 扩展名 | 语言 |
|--------|------|
| .py | python |
| .java | java |
| .js | javascript |
| .ts | typescript |
| .xml | xml |
| .yml / .yaml | yaml |
| .toml | toml |
| .json | json |
| .sql | sql |
| .md | markdown |
| 其他 | unknown |

#### 主要方法

- **`detect(file_path: str) -> str`**：通过 `Path(file_path).suffix` 提取扩展名后查询映射表。  
- **`detect_by_extension(extension: str) -> str`**：直接根据扩展名字符串推断，自动补全点号。  
- **`supported_extensions() -> list[str]`**：返回所有支持的扩展名列表。

#### 设计理由

- 独立模块，不耦合于 `RepoFile`，以后可以无痛升级为基于内容或 shebang 的检测器。
- 使用 `Path.suffix` 代替字符串切割，避免路径中包含点号时的边界错误。
- 全静态方法，无需实例化，调用简单。

## 快照构建器 SnapshotBuilder

**文件位置**：`repoguide/core/snapshot/snapshot_builder.py`

`SnapshotBuilder` 负责将分散的扫描、分类、项目类型识别等步骤**编排成一个完整流程**，并产出 `RepoSnapshot`。

### 静态方法

- **`build(root_path: str, repo_id: Optional[str] = None, project_name: Optional[str] = None) -> RepoSnapshot`**  
  1. 调用已有的 `scan_repo()` 获取原始文件信息。  
  2. 调用 `classify_files()` 为每个文件打上 `role`。  
  3. 通过 `RepoFile.from_dict()` 将字典列表转换为 `RepoFile` 对象列表。  
  4. 调用 `identify_project_type()` 获得项目类型。  
  5. 创建并返回 `RepoSnapshot`（便捷列表由 `RepoSnapshot.__post_init__()` 自动填充）。

### 设计理由

- 将 v0 中手动串联的步骤标准化，任何上层模块（如 CLI、测试）只需调用一个方法即可获得完整快照。
- 不重新实现业务规则（如项目类型识别），完全复用已有组件，保证输出与 v0 一致。
- 路径处理使用 `pathlib.Path`，确保跨平台兼容。

## 项目映射器 ProjectMapper

**文件位置**：`repoguide/core/mapper/project_mapper.py`

`ProjectMapper` 提供了**新旧两套接口**，分别服务于 v0 和 v1 的消费者。

### 保留的旧接口

- **`generate_project_map(files: List[Dict], root_path: str, project_type: str = "unknown") -> Dict`**  
  完全按照 v0 的逻辑工作，返回字典，供 `main.py` 和旧测试使用。

### 新增的 v1 接口

- **`generate_project_map_from_snapshot(snapshot: RepoSnapshot) -> ProjectMap`**  
  1. 调用旧接口获得基础字典。  
  2. 统计 `language` 和 `role` 的分布。  
  3. 计算 `top_languages`。  
  4. 组装并返回 `ProjectMap` 实例。

### 关键辅助

- **`_generate_top_level_tree_from_files(**kwargs**) -> str`**：基于文件路径生成一级目录树，不读取磁盘。
- 模块级便捷函数：`generate_project_map(...)` 和 `generate_project_map_from_snapshot(...)`，方便直接导入。

### 设计理由

- 完全**向后兼容**：v0 调用方无感知升级，旧测试和新测试共存。
- v1 接口提供了更丰富的统计信息，且返回类型安全的 `ProjectMap`，避免了字典键名的脆弱性。
- 逻辑复用：新接口站在旧接口的肩膀上，减少代码重复。

## 核心门面 RepoGuide

**文件位置**：`repoguide/core/repoguide.py`  
**包导出**：`repoguide/__init__.py` 和 `repoguide/core/__init__.py`

`RepoGuide` 是面向外部的高级接口，将核心 SDK 的能力封装为简单的方法调用。

### 静态方法

- **`map(root_path: str) -> ProjectMap`**  
  1. 调用 `SnapshotBuilder.build(root_path)` 生成快照。  
  2. 调用 `generate_project_map_from_snapshot(snapshot)` 生成项目地图。  
  3. 返回 `ProjectMap`。

若 `root_path` 为空或不存在，将抛出 `ValueError` 或 `FileNotFoundError`。

### 使用示例

```python
from repoguide import RepoGuide

pm = RepoGuide.map(".")
print(pm.project_type)      # 'python-project'
print(pm.top_languages)     # ['python', 'markdown']
```

### 设计理由

- **单一入口**：隐藏了内部 Scanner、Classifier、Builder、Mapper 的复杂交互。
- **未来扩展性**：后续可添加 `index`、`ask`、`diagnose` 等方法，而不影响现有调用。
- **易于测试**：只需 mock 底层组件，即可对门面进行集成测试。

## 测试策略

为了保证迁移过程中不引入回归，测试覆盖了两个代际的代码路径。

| 测试文件 | 覆盖内容 |
|----------|----------|
| `tests/test_language_detector.py` | 正常扩展名识别、未知文件、无点号扩展名、支持列表。 |
| `tests/test_repo_file.py` | 语言自动推断、`from_dict`/`to_dict` 桥接正确性。 |
| `tests/test_repo_snapshot.py` | 便捷列表自动填充、去重排序。 |
| `tests/test_snapshot_builder.py` | 集成测试，通过 mock 验证整个构建流程。 |

| `tests/test_project_mapper.py` | v0 接口和 v1 接口的输出对比，确保兼容。 |
| `tests/test_repoguide.py` | 端到端冒烟测试，验证 `RepoGuide.map()` 产出正确。 |

所有测试使用 `pytest` 运行，无需外部依赖。

## 迁移路线与兼容性

当前 v1 Core SDK 遵循“新增不破坏”原则，迁移路线如下：

1. **第一阶段（已完成）**  
   - 定义数据模型 `RepoFile`、`RepoSnapshot`、`ProjectMap`。  
   - 实现 `LanguageDetector`、`SnapshotBuilder`。  
   - 扩展 `ProjectMapper`，新增 v1 接口，保留旧接口。  
   - 提供 `RepoGuide` 门面。

2. **第二阶段（进行中）**  
   - 修改 `main.py` 使用 `RepoGuide.map()`，验证新旧路径输出一致。  
   - 逐步让 formatter 直接接收 `ProjectMap` 对象（当前仍通过 `to_dict()` 桥接）。

3. **第三阶段（计划中）**  
   - 完全移除对旧 dict 接口的依赖，统一使用数据模型。  
   - 基于 `RepoSnapshot` 开发索引器、接口解析器。  
   - 实现 CLI 工具，取代 `main.py` 脚本。
